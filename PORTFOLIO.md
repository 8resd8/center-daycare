# 🏥 요양기록 AI 매니저 - 포트폴리오

## 📖 프로젝트 배경 스토리

주간보호센터에서는 매일 수십 명의 어르신에 대한 장기요양급여 제공기록지를 작성합니다. 이 기록지는 신체활동지원, 인지관리, 간호관리, 기능회복 등 4개 영역의 특이사항을 포함하며, 공단 평가 시 OER(관찰-개입-반응) 구조를 따라야 합니다.

기존에는 수기로 작성된 PDF를 일일이 검토하고, 품질이 미흡한 기록을 찾아 피드백하는 데 많은 시간이 소요되었습니다. 또한 주간 상태변화 보고서를 작성하기 위해 2주간의 데이터를 수동으로 비교 분석해야 했습니다.

이러한 반복적이고 시간 소모적인 작업을 자동화하여 요양 현장 직원들이 어르신 케어에 더 집중할 수 있도록 하는 것이 이 프로젝트의 목표입니다.

## 🎯 문제 정의 → 해결 과정

### 문제 1: PDF 데이터 추출의 어려움

**원인**
- 장기요양급여 기록지는 병합된 셀, 다중 헤더, 별지 등 복잡한 구조
- 한 PDF에 여러 수급자의 기록이 혼재
- 결석/미이용 등 예외 상황 처리 필요

**해결 과정**
1. **pdfplumber 라이브러리 선택**: 테이블 구조 인식에 특화
2. **섹션 헤더 위치 기반 카테고리 할당**: 별지 테이블이 어느 영역(신체/인지/간호/기능)에 속하는지 자동 판별
3. **정규표현식 패턴 매칭**: 날짜, 시간, 수치 데이터 추출
4. **820줄의 파싱 엔진 구현**: 다양한 예외 상황 처리

**결과**
- 10페이지 PDF를 약 3초 내에 자동 파싱
- 별지 내용을 올바른 날짜와 카테고리에 자동 매칭
- 중복 파일 업로드 방지 (SHA-256 해시)

### 문제 2: 특이사항 품질 평가의 주관성

**원인**
- 평가자마다 기준이 다르고, 일관된 피드백 제공이 어려움
- OER(관찰-개입-반응) 구조 준수 여부를 수동으로 확인
- 구체적인 활동명 포함 여부 판단이 주관적

**해결 과정**
1. **프롬프트 엔지니어링**: 요양 현장 전문 용어와 공단 평가 기준을 200줄의 시스템 프롬프트에 반영
2. **3개 후보 생성 전략**: 단일 응답이 아닌 3개 후보를 생성하여 다양성 확보
3. **TF-IDF 유사도 분석**: 이전 문장과 가장 다른 후보를 선택하여 반복 방지
4. **O/X → 점수 변환**: OER충실도, 구체성, 문법 3개 지표를 점수화하여 우수/평균/개선 등급 결정

**결과**
- 객관적이고 일관된 평가 기준 적용
- 개선이 필요한 경우 구체적인 수정 제안 생성
- 날짜별로 다양한 표현의 특이사항 생성 (유사도 기반 선택)

---

### 문제 3: 대량 처리 시 성능 저하

**원인**
- 10명 × 7일 = 70건의 AI 호출을 순차 처리
- 개별 DB 조회로 인한 N+1 쿼리 문제
- 중복 평가로 인한 불필요한 API 호출

**해결 과정**
1. **ThreadPoolExecutor 병렬 처리**: 4개 워커로 동시 처리
2. **배치 DB 조회**: IN 절을 활용하여 N회 → 1회로 감소
3. **캐시 확인**: 이미 평가된 (record_id, category) 조합은 스킵
4. **진행 상황 표시**: Streamlit progress bar로 실시간 피드백

**결과**
- 처리 시간 약 60% 단축 (순차: ~120초 → 병렬: ~45초)
- 데이터베이스 부하 감소
- 사용자 경험 개선 (실시간 진행 상황 확인)

## 🎬 핵심 기능 시연 설명

### 1. PDF 업로드 및 자동 파싱
```
[사용자 동선]
1. 사이드바 "📂 파일 처리" 섹션에서 PDF 파일 업로드
2. 자동으로 테이블 구조 분석 및 데이터 추출 (약 3초)
3. 수급자별로 그룹화하여 목록 표시 (예: "홍길동 · 7건")
4. DB에 자동 저장 완료 메시지 표시

[기술적 하이라이트]
- pdfplumber의 find_tables() 메서드로 테이블 경계 인식
- 섹션 헤더 위치(top 좌표)로 별지 카테고리 자동 판별
- SHA-256 해시로 중복 파일 업로드 방지
```

### 2. 데일리 특이사항 AI 평가
```
[사용자 동선]
1. 수급자 목록에서 평가 대상 선택
2. "데일리 특이사항 평가" 탭 이동
3. "🚀 전체 특이사항 평가 시작" 버튼 클릭
4. 각 날짜별 신체활동/인지관리 특이사항 AI 평가 진행
5. 평가 결과 테이블에 우수/평균/개선 등급 및 수정 제안 표시

[기술적 하이라이트]
- GPT-4o-mini에 OER 구조 평가 프롬프트 전송
- JSON 응답 파싱 (original_*_evaluation, *_candidates)
- TF-IDF 벡터화로 이전 문장과 유사도 계산
- 가장 유사도 낮은 후보 선택하여 다양성 확보
```

### 3. 주간 상태변화 보고서 생성
```
[사용자 동선]
1. "📄주간 상태 변화 평가" 탭 이동
2. 2주간 데이터 자동 비교 분석 결과 확인
   - 식사량 변화율: +5.2% 상승
   - 배설 횟수 변화율: -2.1% 하락
3. "생성하기" 버튼 클릭
4. AI가 작성한 주간 보고서 확인 (약 2초)
5. "복사하기" 버튼으로 클립보드 복사

[기술적 하이라이트]
- Pandas 데이터프레임으로 2주간 집계
- 출석당 평균 계산 (공정한 비교)
- 이전 주 보고서를 AI에 참조시켜 일관성 유지
- JavaScript clipboard API로 복사 기능 구현
```

### 4. 전체 인원 일괄 처리
```
[사용자 동선]
1. 사이드바 "전체인원 AI 처리" 섹션
2. "데일리 특이사항 평가" 버튼 클릭
3. 진행 상황 표시 (예: "🔍 [홍길동] 평가 중... (3/10)")
4. 완료 메시지 확인 (약 45초)

[기술적 하이라이트]
- ThreadPoolExecutor(max_workers=4)로 병렬 처리
- 전체 고객 ID를 IN 절로 한 번에 조회
- Set 기반 캐시로 중복 평가 방지
- 개별 평가 실패가 전체 프로세스에 영향 주지 않도록 에러 격리
```

## 🏗 아키텍처 설계

### 계층 구조
```
┌─────────────────────────────────────┐
│   Presentation Layer (Streamlit)    │
│  - app.py, pages/, modules/ui/      │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│      Service Layer (Business)       │
│  - daily_report_service.py          │
│  - weekly_report_service.py         │
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│   Repository Layer (Data Access)    │
│  - customer.py, daily_info.py       │
│  - weekly_status.py, ai_evaluation.py│
└──────────────┬──────────────────────┘
               │
┌──────────────▼──────────────────────┐
│      Database Layer (MySQL)         │
│  - customers, daily_infos           │
│  - weekly_statuses, ai_evaluations  │
└─────────────────────────────────────┘
```

### 디자인 패턴
- **Repository Pattern**: 데이터 접근 로직 분리
- **Service Layer Pattern**: 비즈니스 로직 캡슐화
- **Context Manager Pattern**: DB 트랜잭션 자동 관리
- **Singleton Pattern**: AI 클라이언트 재사용 (`@st.cache_resource`)
